CHAPTER 01
หลักการพื้นฐาน
ทำไมการตั้งชื่อไฟล์ให้ถูกต้องถึงสำคัญ

ไฟล์งานทุกชิ้นในบริษัทถูกจัดเก็บใน NAS ซึ่งมีหลาย entity หลาย project และหลายคนเข้าถึง การตั้งชื่อไฟล์ที่ถูกต้องทำให้

  • ค้นหาไฟล์ได้ทันทีโดยไม่ต้องเปิดดูทีละไฟล์
  • รู้ทันทีว่าไฟล์นี้เป็นของบริษัทไหน ลูกค้าไหน ประเภทอะไร version อะไร
  • ป้องกันการ overwrite ไฟล์ที่ยังใช้งานอยู่
  • สมาชิกใหม่เข้าใจ structure ได้ทันทีโดยไม่ต้องถาม
  • Archive งานได้อย่างมีระบบ ค้นย้อนหลังได้หลายปี
📌
กฎสำคัญที่สุด: ใช้ underscore (_) แทน space เสมอ ห้ามใช้ภาษาไทยในชื่อไฟล์ ห้ามใช้อักขระพิเศษ เช่น !, @, #, (, ), /, \
CHAPTER 02
Format หลัก
โครงสร้างชื่อไฟล์ที่ใช้กับทุกบริษัท
ทุกบริษัท
MASTER FORMAT
[YYYYMMDD] _ [ENTITY] _ [CLIENT/PROJECT] _ [TYPE] _ [DESCRIPTION] _ [VERSION/SEQ] .[EXT]
ตำแหน่ง ฟิลด์ รูปแบบ ตัวอย่าง หมายเหตุ
1 [YYYYMMDD]
จำเป็น		 8 หลัก ปีเดือนวัน 20260518 วันที่ถ่าย / สร้างไฟล์
2 [ENTITY]
จำเป็น		 2 ตัวอักษร CC / BG / TR รหัสบริษัท ดูบทที่ 03
3 [CLIENT/PROJECT]
จำเป็น		 ชื่อย่อ ไม่เกิน 15 ตัว THAIMART / NIKE ใช้ชื่อย่อที่ตกลงกันไว้ใน Sheet
4 [TYPE]
จำเป็น		 รหัส TYPE PHOTO / VDO ดูบทที่ 04
5 [DESCRIPTION]
แนะนำ		 ชื่อย่อ scene/item LookbookA / Scene03 อธิบาย content ของไฟล์
6 [VERSION/SEQ]
จำเป็น		 v## หรือ ### (3 หลัก) v01 / 001 ไฟล์ project ใช้ v## / ไฟล์ภาพใช้ ###
💡
Version vs Sequence: ไฟล์ที่มีการแก้ไข (edit, retouch, poster) ใช้ v01, v02... / ไฟล์ภาพถ่ายที่เป็น sequence ใช้ 001, 002...

CHAPTER 08
Version Control
การจัดการ version ไม่ให้ไฟล์หาย
ทุกบริษัท
ใช้เมื่อFormatตัวอย่าง
ไฟล์ project / เอกสาร ที่มีการแก้ไข v01, v02, v03... ..._EDIT_v01.prproj → _v02.prproj
ไฟล์ภาพถ่ายที่เป็น sequence 001, 002, 003... ..._001.raw → _002.raw
ไฟล์ส่ง review หลายรอบ REVIEW_v01, v02... ..._REVIEW_v01.mp4 → _v02.mp4
ไฟล์ final ที่ approve แล้ว เพิ่ม FINAL หรือ APPROVED ..._MASTER_FINAL_v01.mov
🚫
ห้ามทำ: ห้ามใช้ชื่อ final_final, final2, new_version, จริงๆ — ใช้ระบบ v## เท่านั้น

_EDIT_v02.prproj
_REVIEW_v01.mp4
_MASTER_FINAL_v01.mov
❌ ห้ามทำ
_EDIT_final.prproj
_EDIT_new.prproj

CHAPTER 10
Do & Don't
กฎสำคัญที่ทุกคนต้องรู้
ทุกบริษัท
✅ ทำ
ใช้ภาษาอังกฤษเท่านั้น
ใช้ _ แทน space
ใช้ตัวพิมพ์ใหญ่สำหรับ TYPE code
ระบุ version ทุกครั้งที่แก้ไข
บันทึกวันที่ขึ้นต้นเสมอ
ระบุ Entity (CC/BG/TR) ทุกไฟล์
ย้ายไฟล์เข้า folder ที่ถูกต้อง
ตรวจ naming ก่อน save ทุกครั้ง
❌ ห้าม
ห้ามใช้ภาษาไทยในชื่อไฟล์
ห้ามใช้ space ในชื่อไฟล์
ห้ามใช้ !@#$%&*()[]{}|/\
ห้ามชื่อ "final_final" หรือ "new"
ห้ามทิ้งไฟล์ไว้ Desktop
ห้ามลบไฟล์โดยไม่แจ้งทีม
ห้าม overwrite ไฟล์เก่า
ห้ามใช้ชื่อ folder ยาวเกิน 30 ตัว
🆘
ไม่แน่ใจ? ถามถั่วก่อน ดีกว่าตั้งชื่อผิดแล้วต้อง rename ทีหลัง การ rename ไฟล์ที่อยู่ใน NAS อาจทำให้ link ใน project files เสียหายได้
